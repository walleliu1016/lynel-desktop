## 0.0.12

### Bug 修复

- **关闭/重启后 `--resume` 找不到 conversation**：`createSessionInternal` 用 `randomUUID()` + `--session-id` 启动时，Claude 的 jsonl 是懒创建的（要发首条 prompt 才落盘），但 lynel 立即把 sid 写进 `recent-sessions.json`，导致用户在发 prompt 前关闭/重启后留下"phantom session"。本次修复：
  - `App.openTerminal` 加 jsonl 存在性检查，缺失时 fallback 到 `PtyMode.New` + 同 sid 重新拉起（jsonl 路径不变），规避 `No conversation found with session ID` 错误
  - `createSessionInternal` 末尾的 `addRecentSession` 改为延迟到 hookserver 收到首个 `UserPromptSubmit` 事件后再写，从源头消除 phantom session 进入 recent 列表

- **企业微信 markdown 消息超长 40058**：企微对单条 markdown 消息硬限制 20480 字节，长 tool result / 长 agent 文本会触发 `errcode=40058 body.markdown.content exceed max length 20480`。`WeComChannel.sendContent` 入口加长度检查，超过 19000 字符（保留 ~1.5KB buffer 应对 JSON 序列化膨胀）时自动截断并标注"内容过长已截断 N 字符"

- **企业微信 `/screenshot` 终端截图渲染错乱**：`renderBufferToPng` 此前有 4 个 bug 导致截图与终端实际显示不一致：
  1. `rawBuffer.replace(/\r/g, '\n')` 把单独 CR 替换为 LF，破坏 spinner/进度条等"行首覆盖"重绘语义
  2. headless xterm 硬编码 `rows: 100`，与 PTY 实际 rows（通常 24）不一致，光标定位序列和 `\x1b[J` 擦除错位产生大片空白
  3. `if (width === 0 || !chars) continue;` 跳过时不推进 `xPixel`，后续 cell 位置偏左，右侧边框被截断
  4. 早期版本试图用 `findLastSgrReset` + `slice(lastReset)` 规避截断颜色泄漏，但会截掉前面所有内容导致截图几乎空白

  修复：直接 `term.write('\x1b[0m' + rawBuffer, ...)`（前置 reset 重置 headless 起始状态），`rows` 接受 `opts.rows` 并从 `session.getSize()` 透传 PTY 实际尺寸，渲染循环分情况推进 `xPixel`（`!cell` 推进 `charWidth`，`width === 0` 跳过宽字符第二列，`!chars` 视作空格推进 `cellPixelWidth`）

### 新功能

- **企业微信发送失败 toast 告警**：`WeComChannel.sendContent` 包裹 `sendFn` try-catch，发送失败时通过 `getBus().emit('app:toast', 'error', ...)` 推送告警到前端，前端 `useEventStream` 已订阅 `app:toast` 事件并 `showToast`，无需前端改动。告警信息包含 sessionId 前 8 位 + 错误信息前 100 字符。仍保留 `throw err` 让外层 `.catch` 不丢感知

- **Cloud 上行 API 接口**：`CloudChannel` 接入两个云服务接口（见 `docs/desktop-interfaces.md`）：
  - `POST /api/envelope/push`：envelope 批量推送，buffer + 3s 定时合并窗口（满 50 条提前 flush），减少 HTTP 请求数
  - `POST /api/sessions/sync`：会话元数据同步，在 session 创建、打开终端、adopt 时触发
  - `app.ts` 新增 `syncCloudSession()` 辅助方法，从 `readRecentSessions()` 取 title 构建 SyncSession，含 `jsonl_path`、`cwd`、`project_name`、`last_activity_at`
  - `CloudTab.vue` 测试连接改为 `POST /api/health`，mock server 同步新增三个端点

- **Hook 事件上行 + PermissionRequest 同步 forward**：`CloudChannel` 实现 `HookChannel`，接入第三个云服务接口 `POST /api/hook`：
  - **非审批 hook**（PreToolUse/PostToolUse/SessionStart/Stop/UserPromptSubmit 等）：fire-and-forget 批量上行，3s/50 条，与 envelope 流量解耦
  - **PermissionRequest**：同步 forward 到云端等待决策（默认 2h 超时），cloud 响应原样透传给 Claude；本地 broker 先 resolve 时 POST `action=abort` 通知云端取消（同时取消本地 fetch）
  - body 统一格式：`{ action: 'forward'|'abort', from: 'desktop', data: <hook body>, request_id?, decision? }`
  - `HookEventLike` 加 `rawBody?` 字段承载 Claude 原始 hook body；`PermissionHandler` 返回类型加 `rawResponse?`，hookserver 存在时原样透传给 Claude
  - `app.ts` onPermissionRequest 入口协调 cloud + broker race：cloud 胜 → 透传 + resolve broker；broker 胜 → cloud abort + 本地 decision；cloud 报错/超时 → 清理后继续等 broker

- **企业微信 `/screenshot` 与 `/help` 控制指令**：
  - 新增 `src/main/terminal-screenshot.ts` 模块：`stripAnsi()` 清洗 SGR/OSC/CUU 等控制码，`renderBufferToPng()` 用 `@xterm/headless` 解析 PTY 原始 buffer + `@napi-rs/canvas` 渲染为 PNG（深色主题，保留 ANSI 颜色与 box-drawing 边框）
  - `wecom-channel` 新增 `__screenshot__` / `__help__` sentinel 路由，`handleScreenshot()` 复用现有会话路由（引用 → bot 绑定 → 默认映射）→ 取 buffer → 渲染 PNG → `uploadMedia` + `sendMediaMessage` 推送到企业微信
  - 失败时回复文本告知原因（bot 未连接 / 会话不存在 / buffer 空 / 渲染失败 / 上传失败），不抛异常
  - 详见 `docs/superpowers/specs/2026-07-23-terminal-screenshot-wecom-design.md`
